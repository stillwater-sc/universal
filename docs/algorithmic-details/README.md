# Algorithmic Details

Deep technical write-ups of selected algorithms used inside Universal. These
docs are for readers who want the math, the transfer functions, and the
trade-off space rather than the API-level usage.

The companion docs to these (API docs, design rationale, decision rationales)
live elsewhere:

- API-level documentation: see the number-system pages under `docs/number-systems/`
- Design rationale (why the framework looks like it does): `docs/design/`
- Per-feature precision/accuracy assessment artifacts: `docs/design/*-precision-assessment.md`

This section is the *educational* layer between those: enough math and
graphical intuition to choose an algorithm or understand a benchmark
result, without the full design-doc machinery.

## Contents

- [LNS log-domain add/sub algorithms](lns-log-add-sub.md) -- the seven
  shipped algorithms for evaluating `log2(1 + 2^d)` and `log2(1 - 2^d)`,
  with transfer-function plots, error envelopes, and a picker decision
  tree.
- [Multi-component floating-point arithmetic](multi-component-arithmetic.md)
  -- Priest's error-free transformations, Bailey/Hida's hand-crafted
  fixed-precision `dd` / `qd`, Shewchuk's adaptive expansions, and how
  Universal's `FloatCascade<N>` building block unifies them.

## Reproducing the plots

The transfer-function and error-envelope figures in these docs are
generated by Python scripts under `scripts/`:

```bash
# from the repo root
python3 docs/algorithmic-details/scripts/generate_lns_plots.py
```

Outputs land in `docs/img/algorithmic-details/`. Rerun whenever the
algorithm parameters or default trait values change.

Dependencies: `python3`, `numpy`, `matplotlib`. The script is intentionally
self-contained: it reimplements the algorithms in Python rather than
binding to the C++ implementations, so the math is visible and the figures
are reproducible without a build.

## Contributing a new entry

Algorithmic-details docs should follow this pattern:

1. State the mathematical problem the algorithm solves, in standard form.
2. Show the *reference* transfer function (the function the algorithm
   approximates) as a plot, with annotations on the singular / limit
   behaviour.
3. Walk through each shipped algorithm: math derivation, runtime
   pseudo-code, error envelope, references.
4. Overlay all algorithms on the reference, plus an error-band panel.
5. Provide a picker decision tree.
6. Cross-link to the implementation, the design doc, the benchmarks, and
   the regression tests.

Keep mathematical notation ASCII per the project CLAUDE.md (`log2`, `<=`,
`->`, `^`, `2^x`).
