## The bfloat16 Fix (#1133 / #1134)

bfloat16 is the high 16 bits of an IEEE-754 float32. The conversion kept those
bits verbatim and dropped the low 16 with no rounding. The fix, in
`convert_ieee754`:

```cpp
// NaN whose payload lives only in the low 16 bits would collapse to +/-inf
if ((bits & 0x7FFFFFFFu) > 0x7F800000u) { _bits = (bits >> 16) | 0x0040u; return *this; }
uint32_t lsb = (bits >> 16) & 1u;   // round-to-nearest-even magic bias
bits += 0x7FFFu + lsb;
_bits = static_cast<uint16_t>(bits >> 16);
```

`0x7FFF + lsb` rounds up on more-than-half, stays on less-than-half, and on an
exact-half tie rounds toward the even significand. Carry into the exponent and
round-to-inf at the range limit both fall out for free. The reported case
`0.2691408770292272` (float bits `0x3E89CCD5`, dropped bits `0xCCD5` > half) now
rounds up to `0.26953125` instead of truncating to `0.267578125`.

## The elreal Cascade (#1135 / #1136)

### The failure

Under ASan/Debug, `el_math_sqrt` aborted:

```
zbcl.hpp:115: ZBCL<bfloat16>::tail(): Assertion `... zero_overlap(...)' && "ZBCL 0-overlap invariant violated"' failed.
```

Instrumenting the assert (print instead of abort) showed exactly one overlap and
then PASS -- the numerical result was correct, only the invariant assert fired:

```
OVERLAP: head exp=-121  tail.head exp=-127  k=7   (bfloat16 emin=-126 => -127 is subnormal)
```

### Why the "correct" fix exposed it

This is the counter-intuitive part. elreal's error-free transforms
(`block_eft.hpp`: `two_sum_host`, Veltkamp/Dekker split) are documented as exact
*only for a round-to-nearest FpType*. The **old truncating bfloat16 was silently
violating elreal's own precondition.** Making bfloat16 conform (RNE) changed the
deep `sqrt` Newton expansion so it bottoms out at the bfloat16 denormal floor,
leaving a trailing **subnormal** limb whose exponent gap to its head is below `k`
(`k = digits = 7`). `block::is_normalised()` documents precisely this:
*"Subnormals fail this predicate; ... 0-overlap accounting assumes the leading bit
is set."*

### The fix, and a red herring

First attempt fixed the streaming `add()` producer (`addRec`) -- it did **not**
help. The leak was the block-list path: `div` returns
`zbcl_from_blocks(priestRenorm(qblocks))` (divide.hpp:127), and `zbcl_from_blocks`
(sum.hpp) only stripped trailing *zero* blocks, not subnormals. The streaming
producers (`mul_scalar`, `online_divide`, `online_multiply`) already dropped
non-normalised blocks; `zbcl_from_blocks` -- the common block-list -> ZBCL choke
point for add/multiply/divide/sum -- did not. One-line fix:

```cpp
while (!blocks.empty() && !blocks.back().is_normalised()) blocks.pop_back();  // subsumes the old zero-block drop
```

Trailing removal is value-preserving because the list is `priestRenorm`'d
(descending exponent), so subnormal limbs sit strictly below every normalised limb.

## Notable Bugs / Gotchas Surfaced

- **`numeric_limits<bfloat16>::digits == 7`**, not 8 (matters for the k-gap).
- **EFT exactness depends on the FpType rounding mode** -- truncating bfloat16 was
  an out-of-contract host all along; the bug was latent, not introduced.
- **The streaming-`add` fix was a dead end** -- always confirm which operation
  actually emits the offending block before patching (instrument, don't guess).
- **`priestRenorm` already documents** narrow-host denormal-floor limbs as benign
  and gates its #1044 extra-pass rescue on `k >= 24`; the `zbcl_from_blocks` guard
  is the consistent complement.
- **Local cross-branch validation trick**: shadow-include the RNE bfloat16 headers
  from the other branch via `-I <dir>` (root is `include/sw`, includes are
  `<universal/...>`), ahead of `-I include/sw`. elreal tests need
  `-DREGRESSION_LEVEL_1=1` to compile.
